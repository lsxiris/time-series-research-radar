## 1. Trigger

Recurring maintenance pass. The repository declares incremental updates as conference and OpenReview sources evolve, and the previous link audit was dated 2026-03-18. Time elapsed since the prior pass made it appropriate to:

- regenerate conference indexes against the current manifests,
- run a fresh public link audit so the README data-quality snapshot stays honest,

- confirm test and validation contracts still hold.

No manifest schema change, no data-source change, and no scope change were introduced. The pass is purely maintenance.

## 2. Agent Decision Tree

The agent followed the maintenance contract in this order, with explicit go/no-go criteria at each step.

```
Step 1: Verify smoke tests
    PASS  -> proceed to Step 2
    FAIL  -> halt; do not regenerate any artifact until tests are restored

Step 2: Verify repository structure
    PASS  -> proceed to Step 3
    FAIL  -> halt; investigate manifest or required-file drift

Step 3: Regenerate conference indexes
    PASS  -> diff is content-stable or content-justified; proceed to Step 4
    FAIL  -> revert generated files; investigate manifest formatting

Step 4: Run public link audit
    PASS  -> capture summary numbers (checked / soft-blocked / failed)
    FAIL  -> if hard failures appear, record them in the audit but do not
              auto-remove the URL; surface the failure for manual review

Step 5: Generate weekly digest sample
    PASS  -> persist as radar/outputs/digests/<ISO-week>.md
    FAIL  -> halt; do not modify other artifacts until digest renderer is fixed

Step 6: Update README, REPO_STATUS, and CHANGELOG with this pass's numbers
    PASS  -> only update fields whose underlying values changed
    FAIL  -> revert; never publish stale numbers as if they were fresh

Step 7: Final review (git status / git diff --stat)
    PASS  -> hand off for human review and commit decision
```

The agent does not commit, push, force-push, or rewrite history during a maintenance pass. The final state is left as a clean, review-ready working tree so the human maintainer can decide whether to bundle the diff into a single commit or split it.

## 5. Risks and Open Items

- The fourteen OpenReview HTTP 429 failures are environmental, not data quality issues. They will most likely return to HTTP 200 the next time the audit runs from a different host or after a longer cooldown. Re-running the audit immediately after a failed pass is not useful — the rate limit window is the bottleneck, not the script.
- The audit script does not currently retry rate-limited URLs with backoff. Adding bounded retry-with-jitter would reduce false-failure noise in future passes.
- `KDD/2025/INDEX.md` and `ICLR/2026/INDEX.md` are the only two indexes whose regeneration produced a real diff. The diff appears to come from upstream manifest field updates, not from a renderer change. A renderer-determinism check (snapshot test on a known manifest) would catch silent renderer changes earlier.
- No schema migration occurred this pass; no manifest field was renamed; no script was changed. If any of these were to occur in a future pass, this log format should add a "schema impact" section.
